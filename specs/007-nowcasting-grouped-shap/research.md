# Research: Nowcasting Grouped SHAP Values

## Decision: Explain only the fitted `phase3_worse` model

**Rationale**: Existing grouped SHAP helpers and forecasting integration are phase-3-specific, and the spec now explicitly scopes the feature to `phase3_worse`. This preserves alignment with existing `forecasting_shap.py` semantics and avoids broadening to all cumulative regressors.

**Alternatives considered**:
- Explain all cumulative targets: rejected because the requirement and evidence support Phase 3+ only, and multi-target SHAP would expand output schemas and tests.
- Make target selectable: rejected as unnecessary scope expansion.

## Decision: Use the exact phase3 training matrix

**Rationale**: The valid SHAP input is equivalent to `train_featured.dropna(subset=["phase3_worse"]).loc[:, feature_columns]`, with columns in fitted-model order. This matches how the phase3 model is trained and satisfies the requirement to avoid prediction-only, validation, or test rows.

**Alternatives considered**:
- Use April prediction rows: rejected because it explains inference data rather than the model as trained.
- Use all training rows before dropping missing `phase3_worse`: rejected because those rows are not necessarily used to fit the phase3 model.

## Decision: Keep grouped SHAP supported only in Mode 1 train-and-predict

**Rationale**: Mode 1 has both the fitted `phase3_worse` model and the corresponding training matrix. Supplied-model compatibility needs provenance and feature-order verification that is out of scope. Prediction-only runs lack model/training context.

**Alternatives considered**:
- Support Mode 2 when compatible: rejected for this feature because compatibility verification increases risk and task scope.
- Best-effort Mode 2 with warnings: rejected because it could produce invalid attribution if the supplied model and rebuilt training matrix differ.

## Decision: Assign `weather forecast` before crosswalk matching

**Rationale**: Runtime forecast proxy features are generated by nowcasting helpers and must not be folded into the original six crosswalk categories. Existing weather proxy definitions such as `forecasted_weather_proxy_columns(scope)` provide conservative authoritative seeds.

**Alternatives considered**:
- Crosswalk first, then weather forecast fallback: rejected because proxy features could be incorrectly assigned to original weather groups.
- Regex-only rainfall/temperature detection: rejected because ambiguous rainfall/temperature features should not automatically become `weather forecast`.

## Decision: Use diagnostics-only unmatched handling

**Rationale**: Existing forecasting SHAP behavior diagnoses unmapped features and excludes them from mapped group denominators. The spec requires transparency and coverage reporting, not an `other` group.

**Alternatives considered**:
- Assign unmatched features to `other`: rejected because it changes group semantics and conflicts with the clarified requirement.
- Drop unmatched features silently: rejected by the spec.

## Decision: Preserve existing forecasting SHAP helpers and add generalized nowcasting extensions

**Rationale**: Existing tests assert six-category behavior, exact matching, and year-by-year matrices for forecasting. New nowcasting behavior needs seven groups and scope columns, so extensions should avoid breaking existing function contracts.

**Alternatives considered**:
- Rewrite `forecasting_shap.py`: rejected because it risks regressions in existing forecasting SHAP.
- Implement all logic only in the CLI wrapper: rejected because reusable code belongs under `src/ipcch/`.

## Decision: One scope per run, combinable scope-by-group outputs

**Rationale**: The nowcasting launch currently runs one `scope_months` value at a time. The output schema should include scope labels and support combining `0m`, `3m`, `6m`, and `12m` into the final heatmap when all are available.

**Alternatives considered**:
- Require one invocation to run all scopes: rejected because the current workflow does not cleanly support it.
- Only single-scope heatmaps: rejected because the user-facing deliverable needs scope comparison.

## Decision: Crosswalk path via explicit override or configured path key

**Rationale**: The constitution requires path-agnostic code. The user-provided Windows path is valid as an explicit override but must not become the only production default.

**Alternatives considered**:
- Hardcode the Windows path: rejected by path-agnostic governance.
- Require path every time: rejected because existing project path mechanisms can provide defaults.
