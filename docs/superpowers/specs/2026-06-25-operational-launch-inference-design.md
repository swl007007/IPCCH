# Operational Launch Inference Design

Date: 2026-06-25

## Purpose

Build a production-ready operational inference path for IPCCH launch predictions.
The research IPCCH repository remains responsible for one-time model training
and model-package export. The `IPCCH_monthly_operational` package is responsible
only for monthly inference from one month of production feature input.

The first model package is prepared for an April 2026 baseline launch. It
exports fixed launch-mode models for three horizons:

- `0m`: feature month predicts the same month.
- `6m`: feature month predicts feature month plus 6 calendar months.
- `12m`: feature month predicts feature month plus 12 calendar months.

The production package must later accept one monthly input file, such as a July
2026 input table, and emit exactly six primary delivery files: one prediction
sheet and one map for each of `0m`, `6m`, and `12m`.

## Confirmed Decisions

- Use the current launch-mode pipeline, not the annual holdout forecasting
  experiment script.
- Do not use forecasted-weather proxy features in the production-default model
  package or monthly inference path.
- Keep model training out of `IPCCH_monthly_operational`.
- Add a production-side `model_pipeline/` directory for pure inference scripts
  and minimal runtime code.
- Add production-side model artifacts under `model_artifacts/`, with separate
  subdirectories for `scope_0m`, `scope_6m`, and `scope_12m`.
- Write monthly delivery outputs under
  `Outcome/ipcch_unified/predictions/YYYYMM/`.
- Treat Docker and cloud deployment as a later phase. This design reserves a
  clean CLI entrypoint and explicit inputs/outputs so containerization is
  straightforward later.

## Out of Scope

- Dockerfile, cloud scheduler, cloud storage orchestration, and managed runtime
  deployment.
- Automated monthly source-data download or feature-building changes.
- Recurring retraining inside the production package.
- The annual `run_deep_feature_weight_decay_forecasting.py` holdout workflow.
- Forecasted-weather inference dependencies.

## Architecture

### Research Preparation Layer

The research repository performs the one-time preparation run:

1. Read the completed historical panel:
   `IPCCH_monthly_operational/Outcome/ipcch_unified/raw/IPCCH_2026_completed.csv`.
2. Read the operational geometry:
   `IPCCH_monthly_operational/Outcome/ipcch_unified/spatial/ipcch_admin_geometry.shp`.
3. Adapt the operational panel to the launch schema by deriving
   `area_id = admin_code` and preserving `admin_code` as a reporting and join
   alias.
4. Generate a feature contract that defines the production-safe feature set and
   compatibility behavior.
5. Train three launch-mode models with scopes `0`, `6`, and `12`, using the
   temporal contract below so April 2026 rows are baseline inference inputs, not
   same-row training labels.
6. Export model weights and metadata to the production package.
7. Generate the April 2026 baseline delivery outputs from the exported package
   to prove the package can run production-style inference.

### Production Inference Layer

`IPCCH_monthly_operational` runs monthly inference only:

1. Read one monthly feature input CSV, normally from
   `Outcome/ipcch_unified/model_input/ipcch_monthly_base_input_YYYYMM.csv`.
2. Validate that the input represents one feature month.
3. Adapt identifiers by deriving or validating `area_id`, with `admin_code` kept
   as a compatibility alias.
4. Apply the model-package feature contract to build a model-ready feature
   matrix.
5. Load fixed model artifacts for `0m`, `6m`, and `12m`.
6. Generate prediction sheets and predicted crisis maps for all three horizons.
7. Write audit reports for feature compatibility, run configuration, and map
   join validation.

## Temporal Training and Inference Contract

The first fixed model package has launch feature month `2026-04`. The April
2026 baseline delivery MUST be generated by using the `2026-04` production
input table as the inference input feature month.

All horizons in a production run use the same input feature month. For the April
2026 baseline:

| Scope | Inference feature month | Prediction target month |
| --- | --- | --- |
| `0m` | `2026-04` | `2026-04` |
| `6m` | `2026-04` | `2026-10` |
| `12m` | `2026-04` | `2027-04` |

The `0m` model training data MUST use only feature months `2026-03` and
earlier. April 2026 rows MUST NOT enter `0m` training. They are reserved as the
baseline inference input rows.

The `6m` and `12m` models MUST be trained with horizon-specific lag/label
alignment. A training sample pairs a historical feature month with its
corresponding future target-month label for that horizon. Inference then applies
the already-trained horizon-specific model to the `2026-04` feature month. The
`6m` and `12m` models do not require April 2026 target labels to train.

No horizon may use information that becomes available only after the allowed
feature-month cutoff for that sample. Training MUST NOT use target-period or
post-target-period information as features. Training MUST NOT use the future
target outcome associated with April 2026 baseline inference rows.

The `0m` business meaning is same-month operational inference / nowcast from a
completed monthly production input table. It is not a month-start forecast. The
monthly input table is normally generated at the end of, or after, the feature
month. For example, a July 2026 production input table is normally available at
the end of July 2026 or later; `0m` predicts July 2026 from that completed July
input and does not promise a July 1 forecast.

`model_pipeline/` contains the pure inference entrypoint and the minimum runtime
code needed to adapt input data, apply feature compatibility rules, load models,
write predictions, and render maps. It must not contain training entrypoints.

`model_artifacts/launch_2026_04/` is the first fixed model package. The package
manifest records source data paths, source hashes where practical, training
cutoff, launch month, horizon list, threshold, package version, runtime
requirements, and whether forecasted weather was used. For this package,
forecasted weather must be recorded as disabled.

## Feature Compatibility Contract

Future monthly inputs may contain only a subset of the columns available in the
2026 completed panel. The model package must therefore include a
`feature_contract.csv` that tells production inference how each model feature is
handled.

Each model feature must have one compatibility category:

- `required`: must be present in the monthly input; missing values above the
  configured tolerance fail validation.
- `derived`: computed by the inference adapter from available input columns.
- `static_join`: joined from a stable package asset keyed by `area_id`.
- `carry_forward`: filled from an approved latest-known source bundled with or
  referenced by the package.
- `median_impute`: filled from training-period statistics stored in model
  metadata.
- `unsupported` or `excluded`: not allowed to be referenced by exported model
  weights.

The `feature_contract.csv` MUST use this machine-executable schema:

| Field | Meaning |
| --- | --- |
| `feature_name` | Model feature name after all preprocessing. |
| `scope_months` | Applicable horizon: `0`, `6`, `12`, or `all`. |
| `category` | One of `required`, `derived`, `static_join`, `carry_forward`, `median_impute`, `unsupported`, or `excluded`. |
| `source_column` | Monthly input column used directly or as a derivation source, when applicable. |
| `dtype` | Expected runtime dtype: `string`, `integer`, `float`, `boolean`, or `categorical`. |
| `required_in_input` | Boolean indicating whether `source_column` must exist in the monthly input. |
| `missing_tolerance` | Maximum allowed missing rate before fill or validation failure. |
| `fill_method` | Runtime fill method: `none`, `median`, `static lookup`, `carry forward`, or `derived`. |
| `fill_value_or_stat_key` | Metadata key or literal value used for imputation. |
| `lookup_asset` | Package-relative path for static or carry-forward lookup assets. |
| `derive_function` | Approved adapter function name for derived features. |
| `as_of_policy` | Rule describing which data timestamp is allowed for this feature. |
| `notes` | Human-readable explanation. |

Source-of-truth rules:

- `feature_columns.json` defines the exact feature order expected by the model.
- `feature_contract.csv` defines how each feature in `feature_columns.json` is
  validated or constructed.
- Every feature in `feature_columns.json` MUST appear exactly once in
  `feature_contract.csv`.
- If a model references a feature missing from the contract, inference MUST
  fail.
- If the contract includes extra features not used by the model, the runtime MAY
  allow them only when they are clearly ignored and reported.
- A feature marked `unsupported` or `excluded` MUST NOT appear in
  `feature_columns.json` or model weights.
- Dtype coercion MUST be deterministic and reported in the compatibility
  report.

Production inference writes both CSV and JSON compatibility reports. These
reports record input columns, model-required columns, filled columns, fill
methods, missing rates, thresholds, and final pass/fail status.

## Production-Safe Feature Eligibility Policy

The training/export stage MUST NOT blindly train on every column in
`IPCCH_2026_completed.csv`. It MUST train only on a production-safe subset.

Features prohibited from exported models:

- Target outcome columns or direct transformations of target outcomes.
- Any column known only after the feature-month cutoff.
- Any feature requiring target-period or post-target-period information.
- Forecasted-weather proxy features, because the production default explicitly
  disables forecasted weather.
- Completed-panel-only fields unavailable in monthly operational input and
  lacking an approved fallback.
- Manually backfilled fields without an explicit as-of policy.
- Rolling, window, lagged, or cumulative features that include information after
  the feature month end.
- Features whose production source, fallback, or static lookup rule is
  undocumented.

Features allowed in exported models MUST satisfy at least one condition:

- Available directly in the monthly production input.
- Derived deterministically from monthly production input fields.
- Stable static attributes joinable by `area_id`.
- Slow-moving or latest-known values with an approved carry-forward policy.
- Features with training-period imputation statistics stored in model metadata
  and allowed by the contract.

Any rolling, lagged, window, or cumulative feature MUST be computed using only
data with timestamp less than or equal to the feature month end. Horizon-specific
label construction MUST NOT change the feature as-of policy. Training/export
MUST include a validation step that rejects future-dependent features before
model training starts.

## Monthly Input Contract

The production input CSV MUST contain exactly one feature month. Required time
columns are `year` and `month`, unless a future schema explicitly replaces them
with a single `feature_period` column.

The CLI `--feature-month` value MUST match the unique `year`/`month` represented
in the input file. If the CLI feature month and input rows disagree, inference
MUST fail.

Identifier rules:

- The input MUST contain either `area_id` or `admin_code`.
- If both `area_id` and `admin_code` exist, they MUST map consistently row by
  row.
- `admin_code` and `area_id` MUST be treated as strings, not numeric values, so
  leading zeros are preserved.
- Each `area_id` MUST appear at most once for the feature month. Duplicate
  `area_id` rows MUST fail validation unless a future aggregation mode is
  explicitly added.

Column and value rules:

- Extra columns are allowed and ignored unless referenced by the feature
  contract.
- Missing values MUST be parsed consistently using documented null tokens.
- Column names are case-sensitive unless the implementation explicitly defines
  a different policy.
- Runtime MUST preserve a stable row identity through validation, feature
  construction, prediction, and map join audit.
- Input does not need to contain every geometry/admin area unless the production
  delivery contract later requires full-universe scoring. Coverage MUST be
  reported.

Input validation happens before model loading and before any primary delivery
file is written.

## CLI Contract

The production command should be non-interactive and future container-friendly:

```bash
python model_pipeline/run_operational_launch_inference.py \
  --input Outcome/ipcch_unified/model_input/ipcch_monthly_base_input_YYYYMM.csv \
  --model-package model_artifacts/launch_2026_04 \
  --spatial-path Outcome/ipcch_unified/spatial/ipcch_admin_geometry.shp \
  --output-dir Outcome/ipcch_unified/predictions/YYYYMM \
  --feature-month YYYY-MM
```

Useful flags:

- `--validate-only`: validate input, model package, feature contract, geometry,
  and output conflicts without prediction.
- `--overwrite`: allow replacement of existing outputs.
- `--no-map`: generate prediction sheets and audit outputs only.
- `--actual-source`: optional actual outcomes for same-month comparison when
  available. Production delivery still counts one map per horizon.

The default mode runs all three horizons in the package. A later implementation
may add `--scopes 0 6 12`, but the initial package should treat all three
deliverables as the normal operational contract.

## Monthly Delivery Outputs

For feature month `YYYY-MM`, with compact label `YYYYMM`, the six primary files
are:

```text
Outcome/ipcch_unified/predictions/YYYYMM/
  ipcch_launch_YYYYMM_scope_0m_predictions.csv
  ipcch_launch_YYYYMM_scope_0m_map.png
  ipcch_launch_YYYYMM_scope_6m_predictions.csv
  ipcch_launch_YYYYMM_scope_6m_map.png
  ipcch_launch_YYYYMM_scope_12m_predictions.csv
  ipcch_launch_YYYYMM_scope_12m_map.png
```

Prediction sheets must include at least:

- `area_id`
- `admin_code` when available
- `feature_period`
- `target_period`
- `scope_months`
- `phase2_worse_score`
- `phase3_worse_score`
- `phase4_worse_score`
- `phase5_worse_score`
- `phase2_worse_pred`
- `phase3_worse_pred`
- `phase4_worse_pred`
- `phase5_worse_pred`
- `overall_phase_pred`
- model package identifier
- source input path or source input hash

Model output semantics:

- `phase2_worse_model` emits a continuous score for the event that outcome
  phase is greater than or equal to 2.
- `phase3_worse_model` emits a continuous score for the event that outcome
  phase is greater than or equal to 3.
- `phase4_worse_model` emits a continuous score for the event that outcome
  phase is greater than or equal to 4.
- `phase5_worse_model` emits a continuous score for the event that outcome
  phase is greater than or equal to 5.
- `phase{k}_worse_score` is the raw model score for phase `k` or worse. It MUST
  NOT be described as a calibrated probability unless model metadata explicitly
  records calibration.
- `phase{k}_worse_pred` is the binary indicator after applying the configured
  threshold to `phase{k}_worse_score`.
- `overall_phase_pred` is the decoded IPC/CH phase derived from the thresholded
  worse-or-equal predictions.

Phase decoding rules:

1. Load thresholds from model metadata. Thresholds MAY be package-level defaults
   or per-scope/per-phase values, but the source MUST be explicit.
2. Apply the configured threshold to each `phase{k}_worse_score`.
3. Produce binary `phase{k}_worse_pred` indicators after thresholding.
4. Decode `overall_phase_pred` from the thresholded worse-or-equal indicators
   using a deterministic top-down rule from phase 5 to phase 2, defaulting to
   phase 1 when no worse-or-equal indicator is active.
5. If worse-or-equal indicators are inconsistent or non-monotonic, the runtime
   MUST either apply a documented deterministic monotonicity correction or fail
   validation. The chosen policy MUST be recorded in model metadata and
   `run_summary.json`.
6. `run_summary.json` MUST record the thresholds actually used.

If compatibility with older launch outputs requires retaining fields named
`phase2_pred`, `phase3_pred`, `phase4_pred`, or `phase5_pred`, the spec treats
them as aliases only. The canonical production fields are the explicit
`phase{k}_worse_score` and `phase{k}_worse_pred` names above.

Maps default to predicted-only crisis maps. If target-period actuals are
available and supplied, the runtime may render an actual-vs-predicted view for
that horizon, but the primary delivery contract remains one map file per
horizon.

Audit files are expected but are not counted among the six primary delivery
files:

- `run_summary.json`
- `feature_compatibility_report.csv`
- `feature_compatibility_report.json`
- `map_join_validation.csv`
- `map_join_validation.json`

## Error Handling

Production inference must fail fast in these cases:

- The input violates the Monthly Input Contract.
- The input contains multiple feature months unless a future batch mode is
  explicitly added.
- A `required` feature is absent or exceeds the contract missing-rate threshold.
- A model references a feature marked `unsupported` or `excluded`.
- A `median_impute`, `static_join`, or `carry_forward` rule lacks the required
  metadata or lookup asset.
- Filled-feature missingness exceeds the configured threshold.
- The shapefile cannot be loaded or contains no usable `area_id` or
  `admin_code` join key.
- Spatial join coverage falls below the configured threshold.
- A primary delivery file already exists and `--overwrite` is not supplied.

Warnings are acceptable for documented fallback use when thresholds are
satisfied. Warnings MUST be machine-readable and appear in both the
compatibility report and run summary.

Fail-fast and partial-output behavior:

- Validation failures MUST NOT create or overwrite any of the six primary
  delivery files.
- If validation fails, the runtime MAY still write diagnostic audit files, but
  they MUST be clearly marked as failed-run artifacts.
- Primary delivery files SHOULD be written atomically: write temporary files
  first, then commit or rename only after all required outputs for the run have
  succeeded.
- If one horizon fails, the whole operational run MUST fail unless a future
  partial-run mode is explicitly added.
- If the run fails after temporary files are created, temporary files SHOULD be
  cleaned up or clearly separated from committed outputs.
- `--overwrite` applies only to committed outputs. It MUST NOT silently mix old
  and new outputs.
- `run_summary.json` MUST include final status, failure reason if any, feature
  month, model package id, scopes attempted, scopes completed, input path or
  hash, and output paths planned or written.
- Compatibility reports and map join validation files MUST include pass/fail
  status and warnings even when thresholds are satisfied.

## Testing Strategy

Unit tests:

- `admin_code` to `area_id` adaptation.
- Single-month input validation.
- Feature contract parsing and enforcement.
- `required`, `derived`, `static_join`, `carry_forward`, `median_impute`, and
  `unsupported` behavior.
- Target-period calculation for `0m`, `6m`, and `12m`.
- Output-path naming and overwrite protection.

Smoke tests:

- Use a small synthetic monthly input and dummy model artifacts to run the full
  production CLI and confirm all six primary delivery paths are planned or
  written.
- Validate that `--validate-only` checks package, input, geometry, and conflicts
  without prediction.

Integration validation:

- Use April 2026 operational input and the exported real model package.
- Run production inference for `0m`, `6m`, and `12m`.
- Confirm prediction schema, feature order, feature compatibility reports, map
  join validation, and output naming match the delivery contract.
- Compare April 2026 production-style predictions with the research export
  outputs to detect feature-order or adapter drift.

## Future Docker and Cloud Readiness

This design intentionally keeps the runtime compatible with later Docker and
cloud execution:

- One non-interactive CLI entrypoint.
- Explicit input CSV, model package, geometry, output directory, and feature
  month parameters.
- All model metadata and feature fallback statistics shipped in the model
  package.
- No training step and no forecasted-weather dependency in the production
  inference command.
- Machine-readable run summaries suitable for cloud logging.

The Docker/cloud implementation should be specified separately after the local
production inference package is verified.
